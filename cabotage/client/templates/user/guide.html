Deploying your code

Every deploy follows a three-step pipeline. You trigger each step from the application's Overview tab.

1
{{ icon('play', 'w-4 h-4 text-primary') }}

Build Image

Compiles your source code into a container image. This runs in a sandboxed build environment and produces a versioned, immutable image.

2
{{ icon('scissors', 'w-4 h-4 text-primary') }}

Create Package

Snapshots the latest image together with your current environment variables into a single deployable unit. Nothing goes live yet — this just locks in what will be deployed.

3
{{ icon('deploy', 'w-4 h-4 text-primary') }}

Deploy

Pushes the package to your pods. Traffic switches over once the new pods pass health checks.

{# ===== SECTION 2: CONFIG-ONLY CHANGES ===== #}
{{ icon('key', 'w-4 h-4 text-primary') }}

Config-only changes

If you only changed environment variables and your code hasn't changed, you can skip the build entirely:

Build → Create Package → Deploy

The package automatically picks up the most recent built image. This is the fastest way to roll out a config change — no rebuild required.

{# ===== SECTION 3: GITHUB AUTO-DEPLOY ===== #}
{{ icon('zap', 'w-4 h-4 text-primary') }}

Setting up auto-deploy from GitHub

Connect a GitHub repository so pushes to a branch automatically trigger image builds.

1

Install the GitHub App

Install the Cabotage GitHub App on your repository or organization.

2

Configure your application

Go to your application's Settings page and fill in:

GitHub Repository e.g. myorg/myapp
GitHub App Installation ID from your org's GitHub App settings
Auto-deploy Branch e.g. main
Subdirectory optional — if your app isn't at the repo root
GitHub Environment Name optional — for custom environment naming
3

Push your code

Push to your configured branch. Once CI passes, Cabotage automatically builds a new image.

4

Package and deploy

You still create packages and deploy manually from the Overview tab. Fully automated deploy can be configured later.

{# ===== SECTION 4: PROCESSES & PROCFILE ===== #}
{{ icon('terminal', 'w-4 h-4 text-primary') }}

Processes & Procfile

Your application's processes are defined in a Procfile at the root of your repository. Each line declares a process type and the command to run it.

Process Types

web

The HTTP-facing process. Receives routed traffic from the load balancer. Must bind to a Unix socket provided by the platform. You can scale this to multiple pods.

worker

Background job processors, queue consumers, or any long-running process that doesn't serve HTTP. Scalable independently from web.

release

A one-off command that runs once during package creation. Use this for database migrations or other setup tasks that need to complete before traffic is routed. The package build will fail if this command exits non-zero.

postdeploy

A one-off command that runs once after each successful deploy. Useful for cache warming, notifications, or cleanup tasks. Failures here do not roll back the deploy.

Example Procfile

web: gunicorn app:app --bind unix:/tmp/nginx.socket
worker: celery -A app.celery worker --loglevel=info
release: flask db upgrade
{{ icon('info', 'w-3.5 h-3.5 text-info/50 mt-0.5 flex-shrink-0') }}

Only web and worker processes are scalable. release and postdeploy always run exactly once.

{# ===== SECTION 5: POD SIZES ===== #}
{{ icon('box', 'w-4 h-4 text-primary') }}

Pod Sizes

Each process runs in a pod with dedicated CPU and memory. Choose a size based on your workload. The naming convention indicates the optimization: m1 is balanced, c1 is CPU-optimized, and r1 is memory-optimized.

Size Type CPU (request) CPU (limit) Mem (request) Mem (limit)
m1.small Balanced 125m 250m 256Mi 384Mi
m1.medium Balanced 250m 500m 512Mi 768Mi
m1.large Balanced 500m 1000m 1024Mi 1536Mi
c1.small CPU 250m 375m 256Mi 384Mi
c1.medium CPU 500m 750m 512Mi 768Mi
c1.large CPU 1000m 1500m 1024Mi 1536Mi
r1.small Memory 125m 250m 1024Mi 1536Mi
r1.medium Memory 250m 500m 1536Mi 2304Mi
r1.large Memory 500m 1000m 2048Mi 3072Mi
r1.xlarge Memory 750m 1500m 2048Mi 3072Mi
r1.2xlarge Memory 1500m 1500m 4096Mi 4096Mi
{{ icon('info', 'w-3.5 h-3.5 text-info/50 mt-0.5 flex-shrink-0') }}

Request is the guaranteed minimum. Limit is the maximum the pod can burst to. CPU is measured in millicores (1000m = 1 core). The default size is m1.large.

Environment variables

Add variables on the Config tab — one at a time, or use the Raw Editor to paste many at once in KEY=VALUE or JSON format.

{{ icon('lock', 'w-3.5 h-3.5 text-warning/60') }} Secret

Stored in Vault. Never displayed in the UI after creation.

{{ icon('eye', 'w-3.5 h-3.5 text-info/60') }} Non-secret

Stored in Consul. Visible and editable on the Config tab.

{{ icon('settings', 'w-3.5 h-3.5 text-success/60') }} Build-time

Available during image build. Useful for private registry tokens.

{{ icon('alert-triangle', 'w-3.5 h-3.5 text-warning/50 mt-0.5 flex-shrink-0') }}

Changes don't take effect until you create a new package. The package snapshots whatever config exists at that moment.
